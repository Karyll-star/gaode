# 高德地图 MCP Server 启动方案

## 目标
- 以 mcprouter 下的高德地图 MCP Server 为数据入口，批量获取候选社区周边的可用 POI/地理结构。
- 将结果整理成“筛选目标信息表与数据库”，字段至少覆盖大致位置、数量、距离等关键指标。
- 按《量化与淘汰标准.md》进行自动评分与淘汰，输出排序清单和淘汰原因，供后续人工复核。

## 输入与输出
- 输入：候选点列表（最少包含名称、城市/区县、可选地址或坐标）。来源可为 CSV/手填。
- 输出：
  - `candidates.csv` 或 `candidates.sqlite`：基础属性+周边统计+评分+淘汰标记。
  - 过程产物：原始 POI 拉取结果 JSON（便于追溯与复跑）。

## 技术路线
1) **定位与范围**：
   - 使用高德地理编码 API（通过 MCP Server 封装）将名称→坐标。
   - 设定多级半径：核心 300m、扩展 500m、观察 800m。用于不同维度的统计。
2) **POI 分类映射**（高德 typecode 仅示例，可在实现时精调）：
   - 高频事务节点 A：
     - 快递/自提：`0601` 邮局、`060102` 快递网点；
     - 餐饮密度：`0500*`；
     - 垃圾/环卫：`190301` 环卫设施；
     - 停车/充电：`1509*` 停车场、`991700` 充电桩；
     - 社区/党群/居委：`141200` 社区服务。
   - 角色混合 B：
     - 学校/幼儿园：`1412*`、`1413*`；
     - 园区/产业/办公：`1701*`；
     - 商业综合体/商铺：`0604*`、`0602*`；
     - 物业/小区信息：`120201` 住宅小区、`190207` 物业公司。
   - 单点试点 C：
     - 门岗/出入口：住宅小区、停车场出入口类 POI；
     - 驿站/快递柜：同快递类；
     - 垃圾点：环卫类；
     - 停车口/车场：停车类。
3) **调用模式**：
   - 通过 MCP Server 的 `place/around`（假定暴露）并行请求上述 typecode；
   - 控制 QPS（建议 10r/s）并做重试；
   - 按半径分桶存储距离信息。
4) **边界/封闭性线索**：
   - 优先获取 `住宅小区` POI 的 `entrance_list`/多点坐标（若接口返回），估算边界清晰度；
   - 辅助：统计围栏类道路（环路/内部道路数量）与出入口数量，粗分“封闭/半开放/开放”。
5) **数据存储**：
   - 采用 SQLite 简单表结构：`targets(id,name,city,lat,lng)`, `poi_hits(target_id,category,typecode,name,lat,lng,distance)`, `scores(target_id,dimension,score,detail)`。
   - 同步生成汇总 CSV 便于阅读。

## 评分与淘汰逻辑（对应《量化与淘汰标准.md》）
- 评分公式：维度分 * 权重，满分 5；权重：节点完整度15%，边界清晰15%，角色混合15%，试点可行25%，可评估性20%，进入性10%。
- 建议量化方式：
  - 节点完整度：A 类命中种类数（≥4→5分，3→4分，2→3分，1→2分，0→1分）。
  - 边界清晰：核心半径内统计“切割线索”（主干道/商超/学校等）；满分 5 分减分制，每命中 1 个切割线索扣 1 分，最低 1 分。
  - 角色混合度：B 类命中种类及数量；≥3 类且均衡→5，2 类→3–4，1 类→2。
  - 试点可行性：C 类节点是否至少 1 个且距离 <300m；多点→5，单点 300–500m→3–4，>500m→2。
  - 可评估性：本阶段不再单独计算，默认 5 分参与加权。
  - 进入性线索：物业/居委/社区服务站命中；有电话/入口备注→5，仅小区 POI→3–4，无→1。
- 硬淘汰：
  - A 类全部缺失；
  - 边界极不清晰且跨多街区；
  - 进入性=1 且无法补充联系方式。

## 流程分解
1. 准备：配置高德 API Key（环境变量 `AMAP_KEY`），确认 MCP Server 运行并在 mcprouter 注册。
2. 导入候选列表：`candidates.csv(name,city,address?)`。
3. 地理编码：对每个候选调用 geocode，存中心点；失败记录待人工修正。
4. 周边检索：按分类批量 around 搜索三层半径，存入 `poi_hits`。
5. 特征聚合：按上表计算各维度原始指标（种类数、平均距离、最小距离等）。
6. 评分与淘汰：套用量化规则写入 `scores` 与最终表。
7. 输出：生成 `report.csv`（含总分、淘汰标记、理由、关键 POI 列表）与 SQLite。
8. 复核：抽样核验 5–10% 点位，人工浏览高德网页版确认。
9. 迭代：根据复核偏差微调 typecode、半径与评分阈值。

## 里程碑与分工示例
- D1：MCP Server 可用性验证、Key 配置、CSV 模版确认。
- D2：完成地理编码与周边拉取脚本；跑通 3 个示例点。
- D3：实现评分模块与硬淘汰逻辑；生成首版 report。
- D4：人工复核+参数调优；确定最终阈值并锁定版本。

## 验收标准
- 至少 10 个候选点跑通，报表字段齐全：坐标、A/B/C 命中数、六维评分、总分、淘汰原因、示例 POI 名称+距离。
- 任一点可复跑，结果相差不超过 5%（数据波动允许）。
- 硬淘汰判定与文档规则一致，无漏判/误保的明显案例。
